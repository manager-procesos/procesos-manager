PRO-MIG-ASP-001 · Procedimiento Operativo · v1.0
Migración de Cliente
Local Manager Time → ASP
Procedimiento formal para la migración de clientes desde instalación local a la nube ASP de Manager. Define responsabilidades, tiempos y entregables para todas las áreas involucradas.
12 etapas 5 áreas involucradas El cliente pierde solo el viernes Mayo 2026

Versión1.0
Fecha de EmisiónMayo 2026
Área PropietariaProyectos / Onboarding
Áreas InvolucradasComercial, GEX, Consultoría Técnica, IT, Onboarding, DTE
EstadoVIGENTE
1
Objetivo
Establecer el procedimiento formal y actualizado para la migración de clientes fidelizados desde la modalidad Local Manager Time hacia la modalidad ASP (Application Service Provider), definiendo las responsabilidades de cada área involucrada, los tiempos de ejecución y los entregables esperados en cada etapa.

Al migrar a ASP, Manager se hace cargo de las actualizaciones de versión y el equipo de soporte puede revisar en línea la base de datos del cliente o bajar respaldos cuando corresponda. El objetivo operativo es que el cliente pierda únicamente el día viernes para el ingreso de documentos al sistema.
2
Alcance
Aplica a todos los proyectos de migración Local → ASP iniciados a partir de la fecha de vigencia. La OT puede originarse desde el Área Comercial (clientes fidelizados) o el Área GEX. Áreas participantes:
Comercial / GEX Proyectos / Onboarding Consultoría Técnica IT Manager Onboarding DTE (si aplica Acepta) Cliente

Glosario
TérminoDefinición
ASPApplication Service Provider. Modalidad en que el sistema opera en servidores remotos gestionados por Manager.
Local Manager TimeModalidad donde el software y la BD operan en los servidores del propio cliente.
BD / Base de DatosConjunto de datos estructurados con la información operativa del cliente.
OTOrden de Trabajo. Documento interno que formaliza el inicio del proyecto.
OnboardingProceso de configuración y validación inicial del cliente en la plataforma ASP.
RamaflexSistema de acceso web para ingresar al entorno ASP desde el navegador.
MaestrosTablas base del ERP: clientes, proveedores, artículos, centros de costo, etc.
Backup / RespaldoCopia de seguridad de la BD en una fecha de corte determinada.
GEXGestor de Expansión. Área encargada de la expansión de clientes con BD de Manager Time instalada.
Mesa de AyudaEquipo de IT de Manager que crea la BD en ASP y emite el correo de bienvenida al cliente.
Fecha de corteFecha y hora exacta en que se genera el backup de la BD, acordada entre el cliente y Onboarding. Idealmente un jueves en la tarde.

Consideraciones Importantes
📅 Fecha de Corte y Backup
La fecha de corte debe acordarse oportunamente con el cliente. El backup debe ser entregado por el IT del cliente en la ruta indicada por Consultoría Técnica, idealmente el jueves PM. Debe quedar registrado por escrito. Consultoría puede apoyar al cliente si se requiere.
🗄 Más de una Base de Datos
Si el cliente tiene más de una BD, se debe indicar en el Checklist Comercial si las BD compartirán el mismo ambiente ASP. Todas las bases deben tratarse en el mismo proceso.
🔍 Validación de Consultoría Técnica
Cualquier inconsistencia detectada en la BD respaldada debe resolverse antes de avanzar a IT. No se carga el backup con datos inconsistentes o incompletos.
🔑 Usuarios y Ramaflex
Es obligatorio verificar que cada usuario tenga correo electrónico válido y vigente. Se debe probar el acceso vía Ramaflex con al menos un usuario por empresa antes de citar al cliente.
🖨 Formatos Personalizados
Los formatos personalizados deben revisarse y adaptarse al entorno ASP en coordinación con Consultoría Técnica. No se puede cerrar el proyecto sin que los formatos estén validados por el cliente.
📄 DTE / Acepta (condicional)
Solo aplica si el cliente tiene contratado Acepta. DTE debe revisar certificados digitales, cesiones de factoring, rutas y parámetros web. Debe participar en la reunión con el cliente y probar la emisión de un documento electrónico en el SII. Si factura con otra entidad, este paso no aplica.
📬 Comunicación con el Cliente
Toda comunicación relevante del proceso (fechas, validaciones, accesos, acuerdos) debe quedar registrada por escrito. El área de Proyectos/Onboarding es la responsable de mantener al cliente informado en todo momento.
