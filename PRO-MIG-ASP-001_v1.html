Objetivo y Alcance
Este procedimiento establece los pasos formales para la migración de una base de datos de cliente desde la modalidad Local Manager Time hacia la modalidad ASP. Define las responsabilidades de cada área, los tiempos de ejecución y los entregables esperados, con el objetivo de garantizar la integridad de los datos y que el cliente pierda el mínimo tiempo posible en el ingreso de documentos al sistema.
⚠
Importante: Este procedimiento aplica a la migración de una sola base de datos. Si el cliente debe migrar más de una BD, los plazos y sesiones de validación con Consultoría Técnica deben coordinarse directamente con el área, ya que cada BD requiere su propio proceso de verificación.
La OT puede originarse desde el Área Comercial o el Área GEX. El Consultor Técnico ejecuta el proceso completo desde la recepción del respaldo hasta el cierre. IT crea el ambiente y carga el backup. DTE participa solo si el cliente tiene contratado Acepta.

Glosario
TérminoDefinición
ASPModalidad en que el sistema opera en servidores remotos gestionados por Manager.
Local Manager TimeModalidad donde el software y la BD operan en los servidores del propio cliente.
BD / Base de DatosConjunto de datos estructurados con la información operativa del cliente.
OTOrden de Trabajo. Documento interno que formaliza el inicio del proyecto.
Consultor TécnicoProfesional de Consultoría Técnica responsable de ejecutar el proceso completo de migración: validación del backup, verificación de la BD, Ramaflex, formatos y cierre.
RamaflexSistema de acceso web para ingresar al entorno ASP desde el navegador.
Backup / RespaldoCopia de seguridad de la BD en una fecha de corte determinada.
Fecha de corteFecha y hora exacta en que se genera el backup de la BD, acordada con el cliente. Idealmente un jueves en la tarde.
GEXGestor de Expansión. Área encargada de la expansión de clientes con BD de Manager Time instalada.
Mesa de AyudaEquipo de IT de Manager que crea la BD en el entorno ASP y emite el correo de bienvenida al cliente.
ChecklistFormulario que completa el área de origen con todos los datos del cliente antes de iniciar la migración.

Consideraciones Importantes
⚠ Una BD por Proceso
Este procedimiento aplica a la migración de una sola base de datos. Si el cliente tiene más de una BD, los plazos y sesiones de validación deben coordinarse directamente con Consultoría Técnica, ya que cada BD requiere su propio proceso de verificación.
📅 Fecha de Corte y Backup
La fecha de corte debe acordarse oportunamente con el cliente. El backup debe ser entregado por el IT del cliente en la ruta indicada, idealmente el jueves PM. Debe quedar registrado por escrito.
🔍 Validación del Backup
Cualquier inconsistencia detectada en el respaldo debe resolverse antes de avanzar a IT. No se carga el backup con datos inconsistentes o incompletos.
🔑 Usuarios y Ramaflex
Se verifica que cada usuario tenga correo electrónico válido. El acceso vía Ramaflex se prueba con el cliente en la misma sesión de verificación.
🖨 Formatos Personalizados
Los formatos personalizados deben revisarse y adaptarse al entorno ASP. No se puede cerrar el proyecto sin que los formatos estén validados por el cliente.
📄 DTE / Acepta (condicional)
Solo aplica si el cliente tiene contratado Acepta. DTE revisa certificados digitales, cesiones de factoring, rutas y parámetros web, y prueba la emisión en el SII en la misma sesión con el cliente.
📬 Comunicación con el Cliente
Toda comunicación relevante del proceso debe quedar registrada por escrito. El Consultor Técnico es el responsable de mantener al cliente informado del avance en todo momento.
